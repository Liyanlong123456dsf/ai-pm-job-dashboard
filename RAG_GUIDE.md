# AI求职助手 — RAG工程搭建指南

## 一、文档用途

本项目包含以下文件，用于在扣子(Coze)平台搭建一个基于真实岗位数据的AI求职助手Agent：

| 文件 | 用途 | 上传位置 |
|---|---|---|
| `knowledge_base.md` | 知识库源文件（1496条岗位） | 扣子 → 知识库 → 上传文档 |
| `coze_prompt.txt` | Agent系统提示词 | 扣子 → Bot → 人设与回复逻辑 |
| `jobs_data.json` | 原始结构化数据（备用） | 本地留存，后续更新用 |

## 二、RAG策略设计

### 2.1 为什么选择Markdown而非CSV/JSON？

| 对比项 | CSV/JSON | Markdown（本方案） |
|---|---|---|
| 分块质量 | 按行切割，上下文断裂 | 按 `##/###` 标题自然分块，每块=1个完整岗位 |
| 检索命中率 | 字段分散，语义匹配差 | 标题+详情+标签在同一块内，语义完整 |
| 层级结构 | 扁平 | 城市→方向→岗位，三级层级，支持范围检索 |
| 扣子兼容性 | JSON不支持直接上传 | Markdown原生支持，自动识别标题层级 |

### 2.2 分块策略（Chunking）

本知识库设计为**标题感知分块**，扣子平台会自动按Markdown标题切分：

```
文档结构:
# AI产品经理岗位知识库          → 全局元信息（不独立成块）
  ## 杭州（280岗，均薪27.5K）   → 城市级块（检索"杭州市场"时命中）
    ### AI产品经理              → 岗位级块（检索具体岗位时命中）
      - 公司/薪资/方向/详情...  → 岗位完整信息在一个块内
```

**每个岗位块包含的可检索字段：**
- 岗位名称（标题语义）
- 公司名称、城市、薪资（精确匹配）
- 方向分类、关键词标签（语义扩展）
- 岗位详情原文（职责+要求，核心检索内容）
- 原始链接（用于推荐时直接跳转）

### 2.3 检索匹配逻辑

```
用户问题 → 向量化 → 在知识库中检索Top-K相似块 → 注入Prompt → LLM生成回答
```

**典型检索场景：**

| 用户问题 | 命中策略 | 示例 |
|---|---|---|
| "杭州AI产品经理薪资" | 城市块+岗位块 | 命中杭州下所有岗位 |
| "大模型方向有什么岗位" | 方向标签匹配 | 命中方向含"大模型/LLM"的块 |
| "3年经验能拿多少" | 经验字段+薪资字段 | 命中经验=3-5年的岗位 |
| "字节跳动在招什么" | 公司名精确匹配 | 命中公司=字节跳动的块 |
| "需要什么技能" | 岗位详情语义匹配 | 命中详情中描述技能要求的段落 |

## 三、扣子平台搭建步骤

### Step 1: 创建知识库
1. 打开 [扣子平台](https://www.coze.cn) → 知识库
2. 点击「新建知识库」→ 命名为 `AI岗位数据库`
3. 上传 `knowledge_base.md`
4. 分段设置：
   - 分段方式：**自动分段**（扣子会按Markdown标题自动分块）
   - 如有手动选项：按 `###`（三级标题）分段，每段=1个岗位
   - 最大分段长度：**800 tokens**（确保单个岗位信息完整）
5. 等待索引构建完成

### Step 2: 创建Bot
1. 扣子 → Bot → 新建Bot
2. 名称：`AI求职助手`
3. **人设与回复逻辑**：将 `coze_prompt.txt` 内容完整粘贴
4. 绑定知识库：选择上一步创建的 `AI岗位数据库`
5. 检索设置：
   - 检索模式：**混合检索**（语义+关键词）
   - 返回条数(Top-K)：**5-8条**（覆盖多角度）
   - 相似度阈值：**0.5**（平衡精度和召回）

### Step 3: 测试与调优
测试问题清单：
```
1. 杭州有哪些AI产品经理岗位？薪资如何？
2. 我3年经验，想转AI方向，推荐去哪个城市？
3. 大模型方向和AI Agent方向哪个薪资更高？
4. 字节跳动在招什么AI产品岗？
5. 经验不限的岗位有哪些？适合应届生吗？
6. 50K以上的高薪岗位都要求什么？
7. 对比一下北京和上海的AI产品经理市场
```

**调优要点：**
- 如果回答太泛泛 → 增加Top-K到8-10
- 如果回答包含不相关岗位 → 提高相似度阈值到0.6
- 如果小城市岗位检索不到 → 检查分块是否完整

## 四、Agent提示词设计逻辑

### 4.1 提示词结构

```
┌─ 角色定义 ─────────────────────────┐
│ "资深AI产品经理求职顾问"           │
│ → 确定专业领域和说话风格           │
├─ 数据描述 ─────────────────────────┤
│ 字段说明 + 数据概况统计            │
│ → 让LLM知道知识库的结构和范围      │
├─ 职责清单 ─────────────────────────┤
│ 5项核心能力：检索/分析/推荐/建议/薪资│
│ → 约束回答方向，避免跑题           │
├─ 回答规则 ─────────────────────────┤
│ 数据驱动 / Markdown / 简洁 / 诚实  │
│ → 控制输出质量和格式               │
└────────────────────────────────────┘
```

### 4.2 关键设计决策

| 决策 | 原因 |
|---|---|
| 提示词内嵌数据概况统计 | LLM无需检索即可回答宏观问题（"一共多少岗位？"） |
| 要求"必须基于数据" | 防止LLM编造不存在的岗位或数据 |
| 限制默认300字 | 控制token消耗，用户需要时再展开 |
| 要求附带链接 | 提升实用价值，用户可直接跳转查看 |
| 11大方向分类写入提示词 | 帮助LLM理解分类体系，精准路由问题 |

## 五、数据更新流程

每日自动更新后，重新生成知识库：

```bash
# 1. 执行每日爬取（自动完成全流程）
python3 scripts/daily_update.py

# 2. 重新生成知识库和提示词
python3 scripts/gen_knowledge.py

# 3. 在扣子平台重新上传 knowledge_base.md
#    （替换旧文档，等待重新索引）
```

## 六、数据字段说明

| 字段 | 类型 | 说明 | 示例 |
|---|---|---|---|
| title | string | 岗位名称 | AI产品经理 |
| company | string | 公司名称 | 字节跳动 |
| city | string | 城市 | 北京 |
| salary | string | 薪资范围(年薪K) | 300-500K |
| avg | float | 等效月薪(K) | 33.3 |
| tier | string | 薪资档位 | 30-50K |
| exp | string | 经验要求 | 3-5年 |
| edu | string | 学历要求 | 本科 |
| cats | list | 方向分类(可多选) | ["大模型/LLM","AI Agent/智能体"] |
| kw | list | 关键词标签 | ["NLP","推荐算法"] |
| desc | string | 岗位详情(职责+要求) | 原文最多400字 |
| url | string | 原始招聘链接 | https://www.zhipin.com/job_detail/xxx |
